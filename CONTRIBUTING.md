# Contributing

这个项目是一个部署到 GitHub Pages 的纯前端页面，目标是保持：

- 手机上可直接使用
- 不依赖后端
- 结构简单、便于维护
- 保质期计算规则稳定可靠

## 项目结构

```text
.
├── index.html
├── styles.css
├── app.js
├── README.md
└── CONTRIBUTING.md
```

- `index.html`
  页面入口，只保留基础结构和资源引用
- `styles.css`
  页面样式
- `app.js`
  页面状态、渲染、交互、日期计算、本地存储

## 开发原则

### 1. 小改动优先

如果只是做视觉优化，优先只改 `styles.css`。  
如果只是改文案，优先只改渲染模板。  
不要在一次提交里混入太多无关修改。

### 2. 保持纯前端

除非明确决定升级架构，否则不要引入：

- 后端服务
- 构建工具
- 复杂依赖
- 框架化重构

当前项目应该继续保持“上传静态文件即可部署”。

### 3. 尽量维持三文件结构

不要轻易把逻辑重新塞回 `index.html`。  
新增功能时优先在现有结构里整理：

- 样式改在 `styles.css`
- 逻辑改在 `app.js`
- HTML 骨架改在 `index.html`

### 4. 输入交互不要整页重绘

搜索框、表单输入框、计算器输入框都容易在手机上失焦。  
修改交互时尽量继续遵守：

- 输入过程优先局部刷新
- 不要每敲一个字就整页 `render()`

## 不能改错的业务规则

这是项目最重要的部分。

### 保质期规则

- 生产日期算第 `1` 天
- 按天计算：
  `过期日 = 生产日期 + 保质期天数 - 1`
- 按月计算：
  先顺延自然月，再减 `1` 天
- 下架日固定为：
  `过期日前 2 天`

示例：

- 生产日期：`2026-03-06`
- 保质期：`7 天`
- 下架日：`2026-03-10`
- 过期日：`2026-03-12`

任何涉及日期逻辑的修改，都必须先确认这条规则没有被破坏。

## 数据语义

### 下架 vs 删除

这两个动作不能混淆：

- `下架`
  代表该批次不再在售，但记录仍然保留
- `删除`
  代表彻底移除该批次记录

如果修改列表逻辑，要确保用户仍能明显区分这两个动作的结果。

### 同名商品分组

当前约定：

- 同名商品按组展示
- 首屏只显示“最快过期的一批”的摘要
- 如果存在多批次，显示轻提示
- 点击商品区域展开批次明细

不要无意中把首页又改回“所有批次直接平铺”的形式。

## 样式修改建议

常见入口：

- 顶部栏：`.topbar`
- 筛选标签：`.tabs`、`.tab`
- 排序按钮：`.sort-row`、`.sort-chip`
- 商品卡片：`.product-card`、`.product-main`
- 日期摘要：`.date-summary`
- 批次明细：`.batch-card`
- 底部导航：`.bottom-nav`
- 悬浮新增按钮：`.fab`

如果只是想“更好看”，优先从这些类开始，不要先改 JS。

## 逻辑修改建议

在 `app.js` 中，优先关注这些函数：

- `calculateBatchDates`
- `groupProductsByName`
- `summarizeProductGroup`
- `sortProductGroups`
- `getVisibleGroups`
- `render`

如果要增加新功能，建议按下面顺序改：

1. 先改数据处理函数
2. 再改渲染函数
3. 最后接事件

## 本地存储

当前使用 `localStorage` 保存数据。  
修改数据结构时请注意兼容旧数据，避免直接导致页面白屏。

建议：

- 读取时做基本校验
- 渲染时对异常数据做兜底
- 不要假设本地存储永远是干净的

## GitHub Pages 注意事项

部署时请保证以下文件位于同一层目录：

```text
repo-root/
  index.html
  styles.css
  app.js
```

不要随意改动相对路径，避免页面资源加载失败。

## 提交建议

推荐提交粒度：

- 一次视觉优化
- 一次交互修复
- 一次日期逻辑调整
- 一次文档更新

避免把“样式大改 + 日期规则修改 + 数据结构调整”混在同一个提交里。

## 推荐工作方式

### 改样式时

1. 先改 `styles.css`
2. 保持 HTML 结构稳定
3. 手机上检查是否有遮挡、失衡、按钮难点的问题

### 改逻辑时

1. 先确认需求没有破坏核心规则
2. 小步修改 `app.js`
3. 特别注意搜索、展开、下架、删除这些高频操作

### 改结构时

如果要新增页面或重构渲染层，先保证：

- GitHub Pages 还能直接打开
- 不引入复杂打包依赖
- 手机浏览器不会因为兼容问题白屏

## 后续建议

如果后续继续迭代，推荐逐步补充：

- `CHANGELOG.md`
- 简单测试脚本说明
- 数据导出 / 导入说明
- UI 设计约定
